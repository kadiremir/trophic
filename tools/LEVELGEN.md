# Trophic Level Generation Tooling

Scripts in `tools/` for generating, validating, and scoring Trophic levels.

---

## Quick start

```bash
# Generate 20 levels and write to src/game/levels.json  (default seed 42)
python tools/generate_levels.py

# Different variety
python tools/generate_levels.py --seed 7

# Validate the existing levels.json without regenerating
python tools/generate_levels.py --validate

# Write to a custom path (e.g. for comparison)
python tools/generate_levels.py --seed 99 --out /tmp/levels_test.json
```

The game reads `src/game/levels.json` directly via `src/game/levels.js`.
**Never hand-edit `levels.json`** — it is the generator's output file.

## Tier structure

| Tier | Label | Chain | Levels | Target range |
|---|---|---|---|---|
| 0 | Fox Forest | G→R→F | 1–5 | 40–60 |
| 1 | Wolf Ridge | G→R→F→W | 6–10 | 60–70 |
| 2 | Bear Mountain | G→R→F→W→B | 11–15 | 70–80 |
| 3 | Dino Peak | G→R→F→W→B→D | 16–20 | 80–90 |

Difficulty targets are explicitly split from **40** (level 1) to **90** (level 20):

```python
TIER_DIFFICULTY_TARGETS = [
    [40, 45, 50, 55, 60],    # Fox Forest
    [60, 63, 65, 68, 70],    # Wolf Ridge
    [70, 73, 75, 78, 80],    # Bear Mountain
    [80, 83, 85, 88, 90],    # Dino Peak
]
DIFFICULTY_TARGETS = [target for tier_targets in TIER_DIFFICULTY_TARGETS for target in tier_targets]
TIER_DIFFICULTY_BANDS = [(tier_targets[0], tier_targets[-1]) for tier_targets in TIER_DIFFICULTY_TARGETS]
```

The band for each biome is a hard acceptance rule. For example, Fox Forest
levels are rejected if their measured difficulty lands below 40 or above 54,
even if they are otherwise close to the target for that slot.

After generation, the accepted levels are sorted by measured `meta.difficulty`
within each biome, then reseated into the final campaign slots. This keeps the
shipped order climbing from easier to harder while preserving the hard biome bands.

## Difficulty formula

```
slack_ratio  = (budget - min_moves) / budget        # 0 = hardest, 1 = easiest
abs_slack    = budget - min_moves                    # raw extra moves granted
solution_count = distinct winning paths (capped at 50)
ambiguity_count = forced-choice points on the first optimal path

difficulty = 100
           - 50 * slack_ratio                        # relative tightness
           - 15 * abs_slack / 5                      # absolute generosity penalty
           - 30 * log(solution_count + 1) / log(51)  # solution breadth
           + 10 * min_moves / 6                      # planning depth bonus
           - 20 / (ambiguity_count + 1)              # forced-choice reward (continuous)
```

**Range:** clamped to [0, 100].

| Term | Max effect | Meaning |
|---|---|---|
| `50 * slack_ratio` | −50 | Move budget generous relative to minimum → easier |
| `15 * abs_slack / 5` | −15 | Absolute extra moves granted → easier |
| `30 * log(sol+1)/log(51)` | −30 | Many winning paths → easier (less planning required) |
| `10 * min_moves / 6` | +10 | Deeper required planning → harder |
| `20 / (ambig+1)` | −20…→0 | Continuous: 0 choices=−20, 1=−10, 2=−6.7, … each extra choice counts |

Constants used: `MAX_SLACK = 5`, `MAX_MIN_MOVES = 6`, `MAX_SOLUTIONS = 50`.

## Agency / quality metrics

These are computed by `score_difficulty` and used as generation filters.

| Metric | What it measures |
|---|---|
| `max_cascade_depth` | Most auto-eat events triggered by a single player move on the first optimal path found |
| `total_auto_events` | Total automatic eats across all moves on that path |
| `agency_ratio` | `min_moves / (min_moves + total_auto_events)` — 1.0 = fully manual, 0.0 = pure cascade |
| `first_move_branching_factor` | Distinct `(sr,sc,tr,tc)` openers that lead to a solution in `min_moves` total |
| `mean_pairwise_distance` | Mean Chebyshev distance between all piece pairs — low = clustered, cascades fire easily |

### Quality thresholds per tier

These are enforced by `_QUALITY_THRESHOLDS` in `generate_levels.py`. A level failing **any** threshold is rejected and a fresh layout is attempted.

| Tier | Max cascade/move | Min agency ratio | Min board spread | Min first-move branching |
|---|---|---|---|---|
| 0 (Fox Forest) | 3 | 0.30 | 1.8 | 1 |
| 1 (Wolf Ridge) | 4 | 0.25 | 2.0 | 2 |
| 2 (Bear Mountain) | 5 | 0.20 | 2.0 | 2 |
| 3 (Dino Peak) | ∞ | 0.10 | 1.5 | 1 |

Tier 3 (Dino Peak) thresholds are intentionally relaxed — the full G→R→F→W→B→D
cascade from a single move is the payoff of that tier, not a defect.

## Generation pipeline

For each of 20 levels (index 0–19):

1. **Tier and target:** `tier = level_index // LEVELS_PER_TIER`,
   `target_diff = DIFFICULTY_TARGETS[level_index]`. Fox Forest now starts at
   difficulty 40, and each biome has its own explicit band (see table above).

2. **Piece counts:** Pick 1–2 complete chains of the tier's food chain, plus optional
   lone grass (for easier levels in tiers 0–2). Dino levels may have 1 or 2 chains,
   but targets in the `85–100` band force **2 full chains** because those slots need
   forced-choice ambiguity to exceed the practical 1-chain ceiling.

3. **Constraint-aware placement:** Places pieces one at a time, choosing only cells
   that don't violate the no-predator-adjacent-to-prey rule with already-placed pieces.
   Falls back to a fresh shuffle on failure; up to 200 inner attempts.  Much faster
   than pure shuffle-and-reject for large piece sets (e.g. dino 2-chain with 12 pieces).

4. **Score objective:** Target = `max(MIN_SCORE_BY_TIER[tier], round(max_achievable * frac))`
   where `frac ∈ [0.45, 0.75]`. Minimum meaningful targets by tier:

   | Tier | Min score target | Rationale |
   |---|---|---|
   | 0 | 4 | At least one full fox chain (G=1 + R=3) |
   | 1 | 12 | At least one wolf chain (+F=8) |
   | 2 | 32 | At least one bear chain (+W=20) |
   | 3 | 40 | At least half a dino chain (+B=48 partial) |

5. **Budget tuning:** `measure_min_moves` is called **once** per layout to get
   `min_moves` and cascade metrics; those metrics are passed into every `score_difficulty`
   call in the scan loop to avoid redundant BFS work. Solution-count / ambiguity probes
   are memoized by effective counting budget while scanning `budget = min_moves` to
   `min_moves + 12`, and the first-move branching factor is deferred until the final
   chosen budget so it is not recomputed during every probe. Hard slack cap `MAX_SLACK = 5`.

6. **Min-moves floor:** Reject if `min_moves < floor`, where:

   | Target difficulty | Required min player moves |
   |---|---|
   | ≤ 20 | 1 |
   | 21–60 | 2 |
   | 61–100 | 3 |

   In practice, the shipped bands now start at 40, so generated campaign levels
   will use only the last two rows of this table.

7. **Biome band gate:** Reject if measured `difficulty` falls outside the active
   biome band:

   | Tier | Allowed difficulty band |
   |---|---|
   | 0 | 40–60 |
   | 1 | 60–70 |
   | 2 | 70–80 |
   | 3 | 80–90 |

   The generator keeps a fallback candidate only if it already satisfies this
   hard band rule, so final output never slips below the intended split.

8. **Progressive tolerance:** Acceptance criteria widen as attempts increase:

   | Attempt phase | Difficulty tolerance | Quality gate |
   |---|---|---|
   | 0–49 % | ±30 | Full (agency + branching) |
   | 50–74 % | ±40 | Full |
   | 75–100 % | ±50 | Relaxed (cascade + spread only) |

   On full exhaustion, the best-found candidate is returned with a `[warn]` log rather
   than hard-failing the run.

9. **Quality gate:** Reject if any agency / spread / branching threshold (see table above) is violated.

10. **Final ordering:** Once all 20 levels are accepted, sort them by measured
    `difficulty` within each tier, then reassign campaign slots in that order.
    IDs, names, hints, and `target_difficulty` metadata are rewritten to match
    the final easier-to-harder sequence. Because tier bands do not overlap, this
    also yields a globally rising campaign difficulty curve.

Up to **200 outer attempts** per level.  Runtime is typically 1–5 minutes for all 20 levels.

## Solvability validation

`measure_min_moves(grid, objective)` — single BFS over `(grid_state, score)` states.
Returns `{min_moves, max_cascade_depth, total_auto_events}` or `None` if unsolvable /
`MAX_SEARCH_STATES = 1_000_000` exceeded.  Call this once per layout and pass
the result as `min_move_metrics=` to `score_difficulty()` to avoid a second BFS.

`find_min_moves(grid, objective)` — thin wrapper returning just the `int` count.
Used by `count_winning_first_moves` which needs a fresh BFS per candidate first move.

`count_solutions` — DFS (node cap 30,000) for solution count + ambiguity count.
The solution count is a lower bound; the node cap may cause undercounting for
complex boards, which inflates measured difficulty slightly.

`count_winning_first_moves` — calls `find_min_moves` once per candidate first move
to count distinct opening `(sr,sc,tr,tc)` choices that lead to an optimal solution.

## Sync checklist — keep engine.py up to date

`engine.py` is a manual port of `src/game/engine.js`.
**Every time `engine.js` changes**, verify or update:

| `engine.js` change | What to update in `engine.py` |
|---|---|
| New entity | Add to `PREY_OF`, `PREY_POINTS`, `JUMP_PRIORITY`; update `TIER_CHAIN` in `generate_levels.py` |
| Food chain link changed | Update `PREY_OF` |
| Forced-choice rule changed | Update `get_forced_jump_options`, `get_forced_choice` |
| Auto-eat cascade rule changed | Update `resolve_jumps` |
| Movement range changed | Update `is_one_step`, `DIRS8` |
| Grid size changed | Update `GRID_SIZE` |
| Scoring changed | Update `PREY_POINTS`, `get_remaining_score_potential` |

After updating `engine.py`:
1. Run `python tools/generate_levels.py --validate` to confirm the existing `levels.json` still passes.
2. If it fails, regenerate: `python tools/generate_levels.py`.
3. Bump `PROGRESS_VERSION` in `App.js` so existing cloud saves are invalidated (see below).

---

## Resetting player progress after regenerating levels

Bump `PROGRESS_VERSION` in `App.js` (e.g. `1` → `2`). Old Firestore saves will
be ignored on next app load and all players start from level 1.
